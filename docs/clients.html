MCP launchers

Same server. Different config files, commands, and scopes.

BrowseFleetMCP is a plain stdio server. Client setup mostly comes down to where the config lives and whether the tool prefers a one-shot CLI command or a checked-in config file.

These snippets only cover the server launcher. Every client still depends on the unpacked Chrome extension from extension-v2/. 

{
  "command": "npx",
  "args": ["-y", "browsefleetmcp"]
}

{
  "command": "node",
  "args": ["/absolute/path/to/browsefleetmcp/dist/index.js"]
}

Codex

CLI install or direct config.toml.

Codex supports both codex mcp add and direct edits to ~/.codex/config.toml. The same MCP config is shared between the CLI and the IDE integration. 

codex mcp add browsefleet -- npx -y browsefleetmcp

[mcp_servers.browsefleet]
command = "npx"
args = ["-y", "browsefleetmcp"]

Cursor

Drop the launcher into ~/.cursor/mcp.json.

Cursor expects the config file directly. Save it, then restart the app so the MCP server list refreshes. 

{
  "mcpServers": {
    "browsefleet": {
      "command": "npx",
      "args": ["-y", "browsefleetmcp"]
    }
  }
}

Claude Code

CLI setup or project-scoped .mcp.json.

Use the CLI when you want user-level setup. Use --scope project if the config should live alongside the repo. 

claude mcp add --transport stdio browsefleet -- npx -y browsefleetmcp

{
  "mcpServers": {
    "browsefleet": {
      "command": "npx",
      "args": ["-y", "browsefleetmcp"]
    }
  }
}

Generic stdio

Any client that accepts command and args.

If the client can spawn a local process over stdio, BrowseFleetMCP fits the standard model. 

{
  "command": "npx",
  "args": ["-y", "browsefleetmcp"]
}

Windows note

Some Windows-native clients need cmd /c in front of npx.

This only matters when the client cannot directly spawn the package manager shim. If it can, the normal npx -y browsefleetmcp entrypoint is still the simplest route. 

{
  "command": "cmd",
  "args": ["/c", "npx", "-y", "browsefleetmcp"]
}
